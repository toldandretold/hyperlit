# Audio — per-node TTS audiobooks ("Listen to this book")

Every row in the `nodes` table gets its own MP3, generated by an open-weight TTS model (Kokoro-82M via DeepInfra's API). Because audio is keyed per node, playback can start from the reader's current position, the paragraph being narrated is highlighted and followed, and an edit invalidates only the audio of the paragraphs that changed (regenerating them costs cents). Generation is requester-pays via credits; the result is playable by everyone RLS lets see the book.

## UX flow

The entry point is the Listen button inside the settings menu (`#audioListenButton` in [resources/views/components/settings-panel.blade.php](../resources/views/components/settings-panel.blade.php), delegated in [resources/js/components/settingsContainer/index.ts](../resources/js/components/settingsContainer/index.ts) → `openAudioPlayer()`). It is `hidden` by default and revealed only on reader pages for narratable books — never for encrypted books or sub-books.

Nothing else is on screen until the button is pressed. If the book has audio, playback starts from the reading position and a small centred pill (`#audio-player-bar` in [resources/views/reader.blade.php](../resources/views/reader.blade.php)) appears at the bottom: play/pause, stop, section counter, speed cycle (1×→2×, client-side `playbackRate`, free), read-along-highlight toggle, a "Resume following" affordance, and a staleness chip. If the book has no audio, a cost-confirm dialog quotes the price (character count × rate × the caller's billing tier); on confirm a background job generates it, the pill shows progress, and the stop button acts as Cancel. Playback begins **before generation finishes**: the client polls and starts as soon as the reader's position has audio (generation runs in reading order), then keeps extending the playlist behind the listener.

## Data model

Migration: [database/migrations/2026_07_06_000001_create_book_audio_tables.php](../database/migrations/2026_07_06_000001_create_book_audio_tables.php). Two tables, RLS enabled + forced, policies cloned from `book_images` (join `library` on book: SELECT = public-or-owner, writes = owner-only).

`book_audio` — one row per narrated node, UNIQUE `(book, node_id)`: `filename` (`{node_id}-{hash8}.mp3`), `source_hash` (**sha256 of the exact `plainText` synthesized** — see Staleness), `voice`, `chars` (billed count), `duration_ms` (estimated from CBR bitrate), `bytes`. Model: [app/Models/PgBookAudio.php](../app/Models/PgBookAudio.php).

`book_audio_meta` — one row per book: pins the `voice` (regenerations must match the original narration), `total_chars`, `generated_by`/`generated_at` (first requester). Model: [app/Models/PgBookAudioMeta.php](../app/Models/PgBookAudioMeta.php).

Bytes live on the private `book_audio` Flysystem disk ([config/filesystems.php](../config/filesystems.php)) at `storage/app/books/{book}/audio/{filename}` — never under `public/`, never symlinked; the RLS-gated route below is the only way to reach them. All file + row writes go through [app/Services/BookAudioStore.php](../app/Services/BookAudioStore.php) (traversal-safe paths, atomic tmp+rename writes, upserts via the BYPASSRLS `pgsql_admin` connection because queue workers have no RLS session, superseded-file deletion, `pruneToNodeIds`, `purgeBook`).

Sizing: 64 kbps mono ≈ 28 MB per hour of narration, so a full novel is ~200–300 MB. The Flysystem seam makes an S3/Spaces swap config-only when volume warrants it.

## Generation pipeline

[app/Jobs/GenerateBookAudioJob.php](../app/Jobs/GenerateBookAudioJob.php), on its own `audio` queue (`npm run queue:audio`, included in `dev:all`; production worker: [deploy/supervisor/hyperlit-audio.conf](../deploy/supervisor/hyperlit-audio.conf) — see deploy/supervisor/README.md, and remember the invariant: a queue nobody listens on silently never runs). Flow:

1. Loads the book's nodes on `pgsql_admin` ordered by `startLine`, keeping the speakable ones. Narration text is derived by [app/Services/Tts/SpeakableText.php](../app/Services/Tts/SpeakableText.php) — **always from `content`, never `plainText`** (plainText is write-path-unreliable — a 280-node prod book had 3 populated rows — and contaminated by bare strip_tags). SpeakableText is a DOM transformer that verbalizes structure instead of leaking it: `[<a class="in-text-citation">9</a>]` → "(citation 9)" (textual citations like "(Smith, 2020)" pass through; the editor's `citation-ref` year unwraps into its "(Author …)" sibling text), footnote markers → "(footnote N)", the hypercite `↗` arrow (all three historical nestings of `.open-icon`, including the AI-archivist's literal `&nearr;` entity that strip_tags used to leak as spoken junk) → "(hypercite link)", empty `latex[data-math]` elements → "equation", `.pageNumber`/`img` dropped, `mark`/`u`/decoration unwrapped, entities decoded, invisible characters (word-joiner etc.) stripped. Headings are read (natural chapter announcements). ⚠ SpeakableText's output IS the `source_hash` input — changing any rule flips every generated book stale (and re-bills its next regen); change deliberately.
2. Prunes `book_audio` rows whose node no longer exists.
3. **Skips every node whose `(node_id, source_hash)` row already matches.** This single rule makes the job idempotent, resumable after a crash, and IS the regenerate-changed-nodes path — "generate" always means "synthesize whatever is missing or stale"; there is no scope parameter.
4. Synthesizes in batches through the provider's concurrent pool (nodes over `max_chars_per_request` are sentence-split and their CBR MP3 segments byte-concatenated); per-node retry ×2; still-failing nodes are recorded in `failed_nodes` and skipped so the book keeps generating.
5. Writes progress beats to `storage/app/books/{book}/audio/audio_progress.json` between batches (with a `stage` field — `preparing` while SpeakableText runs over the book, then `narrating`; the pill shows "Preparing text for narration…" vs the n/m counter) and honours an `audio_cancel` sentinel (the repo's file-based progress idiom — no status columns).
6. Charges **after** generation, for **actually synthesized characters only** ([app/Services/BillingService.php](../app/Services/BillingService.php) `charge`, ledger category `tts`). A partial run bills partially; a retry bills only the gap; no refund logic exists because nothing is charged up front.

Worker RLS gotcha (important if you copy this pattern): `BillingService::charge()` sets only `app.current_user` inside its transaction, but the `users` SELECT policy also requires `app.current_token` — so a queue worker calling it cold silently matches zero rows. The job loads the user via `User::on('pgsql_admin')` and `set_config`s **both** session vars around the charge (see `chargeFor()`).

The TTS provider is a seam: [app/Services/Tts/TtsProviderInterface.php](../app/Services/Tts/TtsProviderInterface.php) (`synthesize` / `synthesizeBatch` / `maxCharsPerRequest`), implemented by [app/Services/Tts/DeepInfraKokoroProvider.php](../app/Services/Tts/DeepInfraKokoroProvider.php) (requests `output_format: mp3` directly — no server-side ffmpeg; `Http::pool` concurrency), selected by `config('services.tts.provider')` in [app/Providers/AppServiceProvider.php](../app/Providers/AppServiceProvider.php). A future self-hosted Kokoro provider (droplet Python service) slots in here and would additionally return word timestamps (`TtsResult.wordTimestamps`) for karaoke-style word highlighting.

Config: the `tts` block in [config/services.php](../config/services.php) — env `TTS_API_KEY` (required), `TTS_PROVIDER`, `TTS_BASE_URL`, `TTS_DEFAULT_VOICE` (default `af_heart`), plus pricing (`provider_cost_per_million_chars` 0.80, `billed_per_million_chars` 1.00 — the raw rate passed to `charge()`, on which the billing-tier multiplier then applies).

## API + serving

Controller: [app/Http/Controllers/BookAudioController.php](../app/Http/Controllers/BookAudioController.php). Routes in [routes/api.php](../routes/api.php) (reads are public + throttled — anonymous readers of public books can play; RLS decides visibility — writes are in the auth group) and [routes/web.php](../routes/web.php) (serving):

| Route | What |
|---|---|
| `GET /api/book-audio/{book}/status` | Counts + a cost estimate priced with the **caller's** tier multiplier (feeds the confirm dialog) |
| `POST /api/book-audio/{book}/generate` | Gate ladder: 401 auth → 422 sub-book → 403 encrypted → 404 RLS-invisible → 402 balance (`canProceed`) → 409 `Cache::lock("book-audio:{book}")` busy → 202 dispatch |
| `GET /api/book-audio/{book}/progress` | Returns `audio_progress.json` |
| `POST /api/book-audio/{book}/cancel` | Touches the cancel sentinel |
| `GET /api/book-audio/{book}/manifest` | `{voice, nodes: {node_id: {filename, duration_ms, stale}}}` — no ordering; the client sequences via its own IndexedDB node order |
| `GET /{book}/audio/{filename}.mp3` | Streams the file. **Authorization IS RLS** (BookMediaController pattern): the row lookup runs on the default connection; invisible → 404, no existence leak. `BinaryFileResponse` handles HTTP Range natively (206), which `<audio>` seeking needs. Public books are CDN/browser-cacheable 1h; private are `no-store` |

Book deletion purges audio via the hook in [app/Services/BookDeletionService.php](../app/Services/BookDeletionService.php); visibility flips need nothing (RLS covers serving + manifest automatically).

## Staleness (stale-but-honest)

There is no stored "stale" flag and no write-path hook. The manifest computes `stale = sha256(SpeakableText::fromContent(nodes.content)) !== book_audio.source_hash` per request, so it is exact truth the moment an edit lands. This deliberately survives the nuclear upsert path (`DbNodeController::upsert` deletes-and-reinserts rows, bumping `updated_at` without content change) — a timestamp scheme would false-positive there. Staleness is surfaced at PRESS-TIME, not on the player: when the Listen button finds existing-but-partially-stale audio, a dialog offers "Update (price) | Listen anyway" (`offerStaleUpdate` in components/audioPlayer/index.ts) — update hits the same generate endpoint (hash-skip = changed-nodes-only + auto-plays when ready), decline plays the current audio. The pill itself carries no staleness chrome.

## Frontend player

Lives in [resources/js/components/audioPlayer/](../resources/js/components/audioPlayer/), registered as a ButtonRegistry component with `pages: ['reader']` in [resources/js/components/utilities/registerComponents.ts](../resources/js/components/utilities/registerComponents.ts) (so it survives SPA nav; never a `@vite` side-effect).

- [index.ts](../resources/js/components/audioPlayer/index.ts) — orchestration: `openAudioPlayer()` entry, Listen-button visibility (`syncListenButton`, also called from settingsContainer after its innerHTML resets), generation watching with play-while-generating, cancel, stale chip.
- [manifest.ts](../resources/js/components/audioPlayer/manifest.ts) — status/progress/manifest fetchers + `audioUrl()`.
- [playbackController.ts](../resources/js/components/audioPlayer/playbackController.ts) — the engine. One persistent detached `HTMLAudioElement`; playlist = IndexedDB node order × manifest; start index = the sessionStorage scroll anchor (searchToolbar pattern); `ended` → next entry (audio-less nodes are simply absent from the playlist); prefetches the next file to warm the HTTP cache; MediaSession metadata + play/pause/prev/next for lock-screen/media-key control; localStorage-persisted settings (speed, highlight, follow).
- [playerBar.ts](../resources/js/components/audioPlayer/playerBar.ts) — the pill view; `.generating` mode hides playback-only controls and repurposes stop as Cancel. CSS: [resources/css/components/audioPlayerBar.css](../resources/css/components/audioPlayerBar.css) (imported by `pages/reader.css`).

Highlight + follow: the playing node gets the `.audio-reading` class; each advance calls `navigateToInternalId(elementId, currentLazyLoader, /*showOverlay*/ false)` ([resources/js/scrolling/internalNav.ts](../resources/js/scrolling/internalNav.ts)), which also **loads the node's chunk if the lazy loader has windowed it out** — that is what keeps follow working across chunk boundaries. Playback itself never depends on the DOM (audio URLs come from the manifest), so narration continues fine even when the narrated paragraph isn't rendered; only highlight/scroll need the element. A user scroll (wheel/touch/scroll-keys) pauses following — narration and highlight continue, the viewport is yours — and following re-engages automatically at the next paragraph boundary after 30 seconds of scroll silence, or instantly via the pill's "Resume following" button.

CSRF note: the generate/cancel POSTs send the `ensureCsrfToken()` cookie token in the **`X-XSRF-TOKEN`** header (`X-CSRF-TOKEN` is the session-token header and 419s with this value).

## Exclusions

- **E2EE encrypted books** (docs/e2ee.md): the server only holds ciphertext (`plainText` is NULL), so server-side TTS is impossible. Gated server-side (`EncryptedBookGuard::isEncrypted` → 403, plus a defense-in-depth check in the job) and client-side (`isBookEncrypted` hides the Listen button).
- **Sub-books / footnotes** (book ids containing `/`): excluded in v1 — 422 server-side, hidden client-side.
- Empty/whitespace `plainText` nodes: never synthesized, never billed, absent from the manifest, skipped in playback.

## Costs

Provider: DeepInfra-hosted Kokoro-82M ≈ $0.80 per million characters. Users are billed $1.00/M raw × their tier multiplier (premium 1.0 ledger-only, budget 1.5, solidarity 2.0, capitalist 5.0). Real-world calibration: a 59k-char paper = 212 nodes ≈ $0.06 raw → $0.09 for a budget user, generated in ~3.5 minutes. A 550k-char novel ≈ $0.55 raw → $0.83 budget. Regenerating ~20 edited paragraphs ≈ $0.02.

## Testing

[tests/Feature/BookAudio/BookAudioTest.php](../tests/Feature/BookAudio/BookAudioTest.php): RLS serving visibility (public/private/anon), Range → 206, the generate gate ladder, manifest staleness after an edit, job idempotency (one charge, second run is a no-op), regen-only-changed-nodes, prune. The TTS provider is faked — no network. Gotchas encoded there: seeded `node_id`s must be book-prefixed (`node_id` is globally unique and admin-seeded residue can collide with other suites), and ledger asserts read the default connection (the charge happens inside the test's uncommitted transaction, invisible to `pgsql_admin`). The test DB needs migrations applied explicitly: `DB_DATABASE=my_laravel_db_test php artisan migrate`.
