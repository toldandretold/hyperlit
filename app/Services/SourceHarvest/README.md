# Source Network Harvester

The Source Network Harvester ("Harvest the Knowledge Commons" in a book's creator tools) builds out the legally-pullable citation network from a starting book: it resolves the book's citations to `canonical_source` rows, then fetches and converts every eligible open-access cited work into that canonical's `auto_version_book` — so the book's citations progressively link to real, held, verified source texts instead of external cards.

## What one run does

A run is one row in `source_network_harvests` driven by `SourceNetworkHarvestJob` (queue `citation-pipeline`, single sequential job, `tries=1`) through `HarvestRunner`. The runner pops `{book, depth}` entries off the row's `frontier` queue and, for each: (1) **scan** — `Artisan::call('citation:scan-bibliography', target)` resolves that book's bibliography/footnote entries against OpenAlex → Open Library → Semantic Scholar and writes `bibliography.canonical_source_id` (the command owns the two-pass bibliography+footnote logic; its results land in DB columns, so no rich return value is needed); (2) **select** — `HarvestEligibility::eligibleCanonicalsFor()` picks the reached canonicals that are unharvested, open access, and fetchable, most-cited first, limited to the run's remaining work budget; (3) **harvest** — `CanonicalVersions\AutoVersionCreator::create()` per work: mint/reuse a system stub, run `ContentFetchService::fetch()` (JATS / OA PDF / HTML / browser lanes), OCR the PDF lane, and wire `canonical.auto_version_book`. Each work is wrapped in try/catch — one bad PDF never kills the run; (4) **shelf** — `HarvestShelf` collects the run's imported sources onto the owner's "Harvested from: <Title>" shelf. On finish the runner bumps `library.annotations_updated_at` on the root book so the frontend re-syncs its bibliography records.

These four stage ids (`scan`, `select`, `harvest`, `shelf`) are the telemetry stage vocabulary and the identifiers in `HarvestMap::stages()` — the single source the live-progress visualisation renders from. `HarvestMapDriftTest` (mirroring the citation pipeline's `PipelineMapDriftTest`, which covers its OWN separate map) fails if a map id stops matching a `HarvestTelemetry::emit()` literal in `HarvestRunner`, or if a `code_ref` stops resolving. Add a stage to the runner = add it to the map.

## Live progress visualisation

`GET /api/source-harvest/map` returns `HarvestMap::stages()`; `GET /api/source-harvest/status/{id}` returns the telemetry event stream (same `{stage, status, detail, signals, at}` shape as the citation pipeline). The frontend overlay `harvestViz.ts` (a port of `aiReview/pipelineViz.ts`, opened from a "See live progress" toggle in the running-state row) renders the four-stage chain with sticky per-stage status, an auto-following details panel, and a completion banner carrying the imported count and the shelf link. The overlay id `harvest-viz-overlay` is focus-trapped via `trapModalFocus` (registered in `overlaySurfacesInventory.json`) — unlike the ai-review viz, which still carries deferred trap debt. Fast 5s polling while the overlay is open, 10s otherwise.

## Email me when done

`POST /api/source-harvest/{id}/notify` (authenticated harvest owner only — anonymous owners have no email — 422 once finished, `throttle:10,1`) sets the `notify_email` flag on the harvest row. `SourceNetworkHarvestJob` then sends `HarvestCompleteMail` / `HarvestFailedMail` (blade views modeled on the import emails; the complete mail carries the counts summary, a link to the book, and the shelf link) from its `handle()` / `failed()` paths — best-effort, a mail failure never fails the job. The frontend "Email me when done" button (in the running row, logged-in users only) POSTs the endpoint and swaps to a confirmation.

## The harvest shelf

`HarvestShelf::ensureShelfFor($rootBook)` find-or-creates a shelf named `Harvested from: <root title>` owned by the root book's creator (the harvest is owner-triggered, so book owner == triggering user), `private` by default; `addBooks()` upserts the run's imported books (`assigned` + `assigned_existing`) and flushes the shelf render cache. Re-harvests of the same book resolve the SAME shelf (keyed on `(creator, name)`) and append — the `shelf_items` PK dedupes. Shelves require a named `creator` (the column is NOT NULL) and only exist on a user page, so anonymously-owned books get no shelf and the shelf step is skipped. Slug generation is shared with `ShelfController` via `App\Services\ShelfSlug` so the two can't drift. The harvest row stores `shelf_id`; the status endpoint returns `harvest.shelf = {id, name, slug, creator}` so the completion dialog and email can link to `/u/{creator}/shelf/{slug}`. CAVEAT accepted by design: two different books with the SAME title share one shelf (uniqueness is `(creator, name)`); the shelf description names the source book.

## The Source Yield Report

Every attempted work's outcome (canonical metadata + status/reason/via/book) is recorded in the harvest row's `results` jsonb — uncapped, unlike telemetry, because the whole point is to keep every failure. At the end of the run (whenever ANY work was attempted, even a 0-import run), `App\Services\SourceHarvest\YieldReportBook::generate()` writes a readable book — the "Source Yield Report" — onto the harvest shelf: an intro summary, a prominent `## Failed to Harvest` section listing each un-fetchable open-access work as a formatted academic citation (title linked to the best source, plus a muted line of the humanised reason — "blocked by the publisher (Cloudflare)", "found a copy but couldn't verify it's the genuine text" — and any other links to try), a secondary `## Harvested` section citing each imported source with its title linked to the held version, and — when a run stopped at its spend cap — a `## Not yet harvested (spending limit reached)` section. Its tone is deliberate: the failures usually aren't beyond a human, only the automation. Citations are formatted by a PHP port of the frontend's `formatMetadataToCitation` (`resources/js/utilities/bibtexProcessor.ts`). The report is built as HTML block-nodes inserted via the raw `pgsql_admin` builder (like `LibraryCardGenerator`/shelf render), which bypasses the `PgNode` saving hook so report text spawns no embedding jobs.

It's ONE LIVING report per book — deterministic id `source-yield-report-<rootBook>`, addressable at `/source-yield-report-<rootBook>` — and it is **CUMULATIVE**: `generate()` keeps a per-book union of outcomes keyed by `canonical_source_id` on the report row's `raw_json.cumulative_results`, merges each run into it (success > failure > skip precedence; a `cloudflare_block` that later succeeds upgrades Failed→Harvested), and rebuilds the nodes from the union. So a later, smaller run — e.g. a second user harvesting the leftovers after a first user's unlimited run — only ADDS or upgrades entries and never clobbers the fuller report. Runs are sequential (the controller 409s a concurrent harvest), so there's no read-modify-write race. The harvest row stores `report_book`; status returns `report_book` + `failed_count`, and `GET /api/source-harvest/latest/{book}` returns the book's latest finished report + shelf so the source panel shows a persistent "See the yield report →" link (mirrors AI Review's "See Review"). Anonymous-owner books get no report (same boundary as the shelf).

## Eligibility

`HarvestEligibility` is the single source of truth. A canonical is eligible when `auto_version_book IS NULL AND is_oa AND (pdf_url or oa_url or doi present)`. Canonicals are reached from the book two ways: directly via `bibliography.canonical_source_id`, and via `bibliography/footnotes.foundation_source → library.canonical_source_id` (footnotes have no canonical column of their own). The same class provides `estimateFor()` — the pure-SQL numbers behind the confirm dialog. Never use `ContentFetchService::dryFetch()` for estimates: despite the name it performs real downloads.

## The fetch ladder (and beating Cloudflare)

`ContentFetchService::fetch()` is the shared acquisition ladder (JATS → ranked OA-copy loop → HTML → browser → FlareSolverr). The tail case — a genuinely-open PDF behind a Cloudflare **managed challenge** — is handled by the browser lane (`scripts/fetch-pdf.mjs` / `fetch-html.mjs` via `scripts/lib/cfBrowser.mjs`): **patchright** (patched Playwright, defeats the CDP-runtime fingerprinting that killed puppeteer-stealth) + a **sticky residential IP** (one session per work, so the challenge solve and the PDF download share the IP that `cf_clearance` is bound to) + a **headed** browser (headless loses managed challenges — needs `xvfb-run` on a server). The PDF is captured with an **in-page fetch inside the cleared session**, not a fresh request that would re-trigger the wall. Config: `SOURCE_FETCH_PROXY` (must be sticky-capable) + `SOURCE_FETCH_STICKY_SUFFIX` + `SOURCE_FETCH_HEADFUL`; see `deploy/oa-fetch-hardening.md`. Iterate/verify one URL with `php artisan harvest:test-fetch {url} --doi=`. When even this loses, it's an honest `cloudflare_block` → the Source Yield Report hands it to a human.

## Frontier and depth (recursion)

Depth is a user choice at trigger time — the `POST .../harvest/trigger` body takes `depth`: `1` (only this book's citations), `2`/`3` (also the works those cite, that many hops out), or `unlimited` (follow the whole reachable open-access network). Every harvest row carries `max_depth` and a `frontier` of `{book, depth}` entries seeded with the root at depth 0. When a work is assigned and `depth + 1 < max_depth`, the new auto-version book is pushed onto the frontier and gets ITS bibliography scanned on a later loop iteration — so the harvester walks the citation network outward. `visited_books` is the cycle guard, and the per-run work budget bounds the total, so `unlimited` terminates when the frontier dries up or the budget is exhausted (it is not literally infinite). The controller maps the choice: depth 1 uses `config('source_harvest.max_works_per_run')` (default 25); deeper/unlimited use `config('source_harvest.max_works_deep')` (default 500); `unlimited` stores `config('source_harvest.unlimited_depth')` (30000, under the signed-smallint ceiling) as `max_depth`. Mind the cost on deep runs: each level multiplies OCR spend and each text's bibliography can be 100+ entries.

## The referenced_works closed pool (scan Wave 3.5)

Every harvested text is by construction linked to a canonical with an `openalex_id`, and OpenAlex publishes each work's outbound citations as `referenced_works`. The scan job exploits this: before the open-ended title-search waves, `CitationScanBibliographyJob`'s Wave 3.5 pulls the parent work's referenced-works ids (`WorksApi::fetchReferencedWorkIds`), batch-loads them (`WorksApi::fetchByIdsBatch`, 50-id `ids.openalex` filters), and scores the still-unresolved bibliography entries against that closed candidate pool with the normal `metadataScore` gates. Cheaper and more precise than open search — exactly what a noisy OCR'd bibliography needs — and entries the pool misses fall through to the existing waves unchanged. Books without an OpenAlex identity skip the wave entirely.

## Budget and idempotency

`max_works` (config `source_harvest.max_works_per_run`, snapshotted onto the row at trigger time) is a hard cap per run; most-cited works are harvested first and the overflow is recorded in `counts.capped`. Re-running after a cap, crash, or failure is cheap and safe: eligibility excludes canonicals whose pointer is already set, `AutoVersionCreator` wires pointers from previous partial runs without fetching (assign-first), `SystemVersionMinter::findExistingSystemRow` reuses stubs, and the scan job skips already-linked entries.

## Cost, the spend cap, and cancellation

Each fetched PDF is OCR'd, which costs real money (Mistral, $1/1k pages). The harvest bills the **owner** pay-as-you-go via `BillingService::billOcrForBook` — the SAME method the document-import path uses, so both share the idempotency marker (`ocr_charged.json`), the native/BYO free-path skip (`hyperlit-` model prefix ⇒ $0), and per-page pricing. `HarvestRunner::chargeWorkOcr` charges after each `assigned` work (a fresh fetch+convert; `assigned_existing` wired an existing stub, no new OCR, no charge). It sets BOTH `app.current_user` and `app.current_token` first (worker RLS — mirrors `GenerateBookAudioJob::chargeFor`). Native OCR (the dev Mac) ⇒ $0; only prod-with-Mistral bills. Anonymous-owner harvests are never billed.

The approve form shows a **rough** cost (`SourceHarvestController::estimate` returns `cost.estimated_user` = eligible × `config('source_harvest.avg_pages_per_work')` × the OCR rate × the caller's tier multiplier — a real page count is unknowable until each PDF is OCR'd) and a **max-spend** field. `trigger` stores it as `max_spend` (null = no cap, balance is the only limit). The runner tracks cumulative `spend` in-process (derivable from `billing_ledger`, so not a column) and, at each work boundary, stops gracefully once `spend >= max_spend` OR the pay-as-you-go balance is exhausted (`canProceed`) — the remaining eligible works are recorded as `skipped_over_budget` (→ their own "Not yet harvested (spending limit reached)" section in the yield report), the shelf + report still finalize, and the run ends `completed`. The cap is a *stop-after* threshold: the work that crosses it may tip slightly over, since page counts aren't known pre-OCR. Premium owners aren't per-use billed, so the cap doesn't apply to them.

**Cancellation:** `POST /api/source-harvest/{id}/cancel` (owner-gated, 422 once finished) sets `cancel_requested`; the runner polls it at the frontier-loop and per-work boundaries, stops, finalizes the shelf + report for the partial run, and `run()` returns `'cancelled'` so the job stamps status `cancelled`. The frontend running-row has a "Cancel harvest" button; the poll handler treats `cancelled` like `completed`.

## HTTP seam and authorization

`SourceHarvestController`: `POST /api/library/{book}/harvest/estimate` (pure SQL + rough cost), `POST /api/library/{book}/harvest/trigger` (`BillingService::canProceed` + `EncryptedBookGuard` 422 + cache-lock, 409 while a harvest OR a citation pipeline is active for the book), `POST /api/source-harvest/{id}/cancel` + `/notify` (harvest-owner, by `user_id`), `GET /api/source-harvest/status/{id}` / `running/{book}` / `latest/{book}` (polls + report link, with the stale watchdog: pending > 5 min or running > 3 h auto-fails). The job writes everything through `pgsql_admin`, so the controller check is THE authorization boundary.

**Access — owner OR any logged-in user on a commons book.** estimate/trigger use `ResolvesBookOwner::authorizeCommonsWorkflow` (not the owner-only `authorizeBookEdit`): the book's owner may always run it (user or anon token, unchanged), AND any logged-in user may run it on an owner-less **commons book** (`isCommonsBook`: `creator === AutoVersionResolver::CREATOR` or a `SYSTEM_CONVERSION_METHOD`) — requester-pays, everyone benefits, mirroring the audiobook model. Visibility is the RLS read (a private book invisible to a non-owner → 404); a guest on a commons book → 401; a logged-in non-owner on someone else's OWNED book → 403. The **AI Citation Review** trigger (`CitationScannerController::triggerPipeline`) uses the same gate (which also closed a latent leak — it previously checked existence on the RLS-bypassing admin connection). For a commons-book run the shelf + yield report are keyed to the book's system creator and made **public** (a shared, cumulative artifact), so every user's run enriches the same report.

## Commons quality feedback

Auto-harvested commons books have no owner to flag a bad conversion, so the source panel shows a notice (`resources/js/components/sourceContainer/commonsFeedback.ts`) — "converted automatically… report an issue" — that POSTs to the existing `IntegrityReportController::conversionFeedback` (`footnotes_not_matched` / `citations_wrongly_matched` / …) which mails maintainers with `audit.json`/`assessment.json`. Honest framing: it feeds a converter-fix + batch-reconvert loop, not an instant self-heal (re-running the same OCR cache through unchanged code yields the same result).

## Why not citation_pipelines

Harvest state deliberately lives in its own `source_network_harvests` table with its own `HarvestTelemetry` (a clone of `PipelineTelemetry`'s capped read-append-write stream). The citation pipeline's `PipelineMap` + `PipelineMapDriftTest` contract hard-codes its four-stage chain, and a harvest has a different lifecycle (frontier, depth, work budget). Do not merge them.

## Frontend

`resources/js/components/sourceContainer/creatorTools/harvestNetwork.ts` renders the button (a monochrome combine-harvester icon from `combineIcon.ts` + a "?" info expandable), runs estimate → `choiceDialog` (the depth picker — this book only / one or three levels deeper / unlimited) → trigger with the chosen `depth`, polls status every 10s (button text mirrors `step_detail`), and calls `refreshCitationDisplay()` on completion. No changes to `displayCitations/` were needed: matched references already resolve through `/api/canonical/{id}/best-version`, which prefers held versions including `auto_version_book`.
